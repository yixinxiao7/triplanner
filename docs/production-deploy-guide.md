# TripPlanner — Production Deploy Guide

**Target environment:** Render (frontend + backend) + AWS RDS PostgreSQL 15
**Written:** 2026-03-11 (Sprint #26, T-222)
**Prerequisites:** Complete T-220 (knexfile SSL config) and T-221 (cookie SameSite fix) before deploying.

---

## Overview

The production stack consists of:
- **Backend**: Render free-tier web service (Node.js, Ohio region) — `triplanner-backend-sp61.onrender.com`
- **Frontend**: Render free-tier static site (React SPA, Ohio region) — `triplanner.yixinx.com` (custom domain)
- **Database**: AWS RDS PostgreSQL 15, `db.t3.micro`, `us-east-1` (free tier)

All services communicate over HTTPS. Render provides HTTPS automatically. The frontend and backend are cross-origin; the `SameSite=None; Secure` cookie fix (T-221) is required for refresh token auth to work.

## Step 1 — AWS RDS: Create the Database

### 1.1 Sign in to AWS Console
Go to https://console.aws.amazon.com and navigate to **RDS**.

### 1.2 Create a PostgreSQL 15 instance
1. Click **Create database**
2. Select **Standard Create**
3. Engine: **PostgreSQL** — Version: **PostgreSQL 15.x** (latest 15.x patch)
4. Template: **Free tier**
5. Settings:
   - DB instance identifier: `triplanner-production`
   - Master username: `triplanner`
   - Master password: Generate a strong password and save it securely
6. DB instance class: `db.t3.micro` (free tier eligible)
7. Storage: `20 GiB gp2` (free tier default — do not enable autoscaling for free tier)
8. Connectivity:
   - VPC: default VPC
   - **Public access: Yes** (required so Render's egress IPs can connect)
   - VPC security group: Create a new security group named `triplanner-rds-sg`
9. Database name (initial): `triplanner`
10. Click **Create database**

### 1.3 Configure the Security Group
After the instance is created:
1. Go to the RDS instance → **Connectivity & security** → click the security group link
2. Edit **Inbound rules** → Add rule:
   - Type: PostgreSQL
   - Port: 5432
   - Source: **0.0.0.0/0** (Render uses dynamic IPs — open to all, rely on DB credentials for security)
   - Description: `Render egress`
3. Save rules

> **Security note:** Restricting to specific CIDRs is preferable but Render free-tier uses dynamic egress IPs. For stricter security, consider Render's paid tier with static IPs, or use Render's managed PostgreSQL instead.

### 1.4 Note the connection details
From the RDS console, note:
- **Endpoint**: e.g., `triplanner-production.xxxxxxxxxxxx.us-east-1.rds.amazonaws.com`
- **Port**: `5432`
- **Database name**: `triplanner`
- **Username**: `triplanner`
- **Password**: (saved in Step 1.2)

Build the `DATABASE_URL`:
```
postgres://triplanner:PASSWORD@ENDPOINT:5432/triplanner
```

## Step 2 — Render: Connect the Repository

### 2.1 Create a Render account
Go to https://render.com and sign in or create a free account.

### 2.2 Connect GitHub
Go to **Account Settings → Git Providers** → Connect GitHub → Authorize Render.

### 2.3 Deploy via Blueprint (render.yaml)
1. In the Render Dashboard, click **New → Blueprint**
2. Select the `triplanner` GitHub repository
3. Render detects `render.yaml` in the project root and previews the two services:
   - `triplanner-backend` (web service)
   - `triplanner-frontend` (static site)
4. Click **Apply** — Render creates both services

> The first deploy will fail because `sync: false` env vars have not been set yet. That is expected. Complete Step 3 before re-deploying.

## Step 3 — Environment Variables

Set these variables in the Render dashboard for each service. Go to each service → **Environment** tab.

### Backend service (`triplanner-backend`)

| Variable | Value | Notes |
|----------|-------|-------|
| `NODE_ENV` | `production` | Auto-set by render.yaml |
| `DATABASE_URL` | `postgres://triplanner:PASSWORD@ENDPOINT:5432/triplanner` | From Step 1.4 |
| `JWT_SECRET` | *(auto-generated by Render)* | Render generates on first deploy |
| `JWT_EXPIRES_IN` | `15m` | Auto-set by render.yaml |
| `JWT_REFRESH_EXPIRES_IN` | `7d` | Auto-set by render.yaml |
| `CORS_ORIGIN` | `https://triplanner.yixinx.com` | Custom domain for the frontend |
| `PORT` | `10000` | Auto-set by render.yaml; Render overrides with its own PORT |

**Setting `DATABASE_URL`:**
1. Go to `triplanner-backend` → Environment
2. Find `DATABASE_URL` (marked as "Sync: Off — Set manually")
3. Paste the connection string from Step 1.4
4. Click **Save Changes**

**Setting `CORS_ORIGIN`:**
The frontend URL is `https://triplanner.yixinx.com` (custom domain). Set this to match the domain users access.

### Frontend service (`triplanner-frontend`)

| Variable | Value | Notes |
|----------|-------|-------|
| `VITE_API_URL` | `https://triplanner-backend-sp61.onrender.com/api/v1` | Must match the backend service URL |

**Setting `VITE_API_URL`:**
1. Go to `triplanner-frontend` → Environment
2. Find `VITE_API_URL` (marked as "Sync: Off — Set manually")
3. Set value to `https://triplanner-backend-sp61.onrender.com/api/v1`
4. Click **Save Changes**

## Step 4 — Database Migrations

Run `knex migrate:latest` against the production RDS instance **before** the backend handles any traffic.

### Option A: Local migration (recommended for first deploy)

From your local machine with the `backend/` directory:

```bash
# Set DATABASE_URL to point to RDS (temporarily)
export DATABASE_URL="postgres://triplanner:PASSWORD@ENDPOINT:5432/triplanner"
export NODE_ENV=production

cd backend
npm install

# Run all 10 migrations (001–010)
npx knex migrate:latest --knexfile src/config/knexfile.js

# Verify migration status
npx knex migrate:status --knexfile src/config/knexfile.js
```

Expected output: All 10 migration files listed as "Ran".

### Option B: Render shell (after deploy)

If the backend service is deployed, use the Render shell:
1. Go to `triplanner-backend` → **Shell**
2. Run:
   ```bash
   npx knex migrate:latest --knexfile src/config/knexfile.js
   ```

> **Important:** Only 10 migrations need to run (001–010). Sprint 26 has no new schema changes. Do not run `knex seed:run` in production — seeds are for staging only.

## Step 5 — Trigger Deployment

### 5.1 Deploy the backend
1. Go to `triplanner-backend` → **Deploys**
2. Click **Deploy latest commit** (or it may auto-deploy after env vars are set)
3. Monitor the deploy log — look for:
   ```
   [Server] Starting TripPlanner backend on port ...
   ```

### 5.2 Deploy the frontend
1. Go to `triplanner-frontend` → **Deploys**
2. Click **Deploy latest commit**
3. Monitor the build log — look for `vite build` completing without errors

### 5.3 Verify both services are online
- Backend: `https://triplanner-backend-sp61.onrender.com/api/v1/health` → `{"status":"ok"}`
- Frontend: `https://triplanner.yixinx.com` → React SPA loads

> **Note on free-tier cold starts:** Render free-tier services spin down after 15 minutes of inactivity. The first request after a cold start may take 30–60 seconds. This is expected behavior on the free plan.

## Step 6 — Post-Deploy Verification Checklist

Run these checks after both services are online:

```bash
BACKEND="https://triplanner-backend-sp61.onrender.com"
FRONTEND="https://triplanner.yixinx.com"

# 1. Health check
curl -s "$BACKEND/api/v1/health"
# Expected: {"status":"ok"}

# 2. HTTPS enforced (should not get a connection error)
curl -I "$BACKEND/api/v1/health"
# Expected: HTTP/2 200

# 3. Register a test user
curl -s -X POST "$BACKEND/api/v1/auth/register" \
  -H "Content-Type: application/json" \
  -d '{"name":"Deploy Test","email":"deploy-test@example.com","password":"TestPass123!"}' \
  -c /tmp/cookies.txt
# Expected: HTTP 201, {"data":{"user":{...},"accessToken":"..."}}

# 4. Login
curl -s -X POST "$BACKEND/api/v1/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"deploy-test@example.com","password":"TestPass123!"}' \
  -c /tmp/cookies.txt -b /tmp/cookies.txt \
  | jq '.data.accessToken' > /tmp/token.txt
cat /tmp/token.txt
# Expected: JWT string

# 5. Create a trip
curl -s -X POST "$BACKEND/api/v1/trips" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $(cat /tmp/token.txt | tr -d '"')" \
  -d '{"name":"Deploy Test Trip","destinations":["Tokyo"]}'
# Expected: HTTP 201, trip object

# 6. Frontend loads
curl -s -o /dev/null -w "%{http_code}" "$FRONTEND"
# Expected: 200

# 7. Verify SameSite=None cookie in production login response
curl -si -X POST "$BACKEND/api/v1/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"deploy-test@example.com","password":"TestPass123!"}' \
  | grep -i "set-cookie"
# Expected: refresh_token=...; HttpOnly; SameSite=None; Secure
```

### Manual browser checklist:
- [ ] Navigate to `https://triplanner.yixinx.com` — page loads without JS errors
- [ ] Register a new account — redirects to home page
- [ ] Create a trip — appears in trip list
- [ ] Open trip details — flights / stays / activities sections visible
- [ ] View TripCalendar — calendar grid visible at top of trip details
- [ ] Log out — redirects to login

## Rollback Procedure

If production breaks after deploy:
1. In Render → service → **Deploys** → find the last successful deploy
2. Click **Rollback to this deploy**
3. Render redeploys from that commit within ~2 minutes
4. If DB migration caused issues: restore from RDS automated backup (enabled by default on RDS)
